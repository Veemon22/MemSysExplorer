# Results

This directory contains all outputs from end-to-end pipeline runs. Each run produces its own automatically numbered subdirectory so previous runs are never overwritten.

```
results/
└── <config_name>_<N>/
    ├── apps_output/        # pattern config JSONs from profiling
    ├── tech_output/        # NVSim result YAMLs from array characterization
    └── model_output/       # final results CSV
```

`<config_name>` is taken from the name of the run YAML passed to `run.py`, and `<N>` increments by 1 each time you run the same config. For example, running `configs/sniper_run_config_new/sniper_new_run.yaml` twice would produce `sniper_new_run_1/` and `sniper_new_run_2/`.

---

## Subdirectory Contents

### `apps_output/`

Contains the pattern config JSON file(s) generated by the profiler. Each JSON captures the memory access characteristics of a workload — read and write counts, hit/miss counts (Sniper), execution time, and benchmark name — and is what the analytical model consumes downstream. If profiling was skipped (`run: existing`), this directory is not created and the pipeline reads directly from the path provided in `patternconfig_path`.

### `tech_output/`

Contains the NVSim result YAML file(s) from array characterization. Each file is named using the `OutputFilePrefix` from the tech config with the capacity and optimization target appended (e.g. `sample_FeFET_32nm_result_128KB_WriteEDP.yaml`). Each result YAML is fully self-describing — it embeds the memory cell type, capacity, and optimization target it was produced for alongside all timing, energy, power, and area results. If array characterization was skipped (`run: existing`), this directory is not created.

### `model_output/`

Contains the final results CSV: `<config_name>_results.csv`. Each row represents one combination of workload × memory technology × capacity. All runs from the same config file append to the same CSV, so a multi-tech-config or multi-capacity sweep produces multiple rows in a single file.

---

## Example Outputs

The following example run directories are included in `results/` as reference outputs. Each shows what a complete pipeline result looks like for a specific configuration and can be used to verify your setup is producing expected results before running your own workloads.

- **`sniper_new_run_1/`** — cache run using Sniper on a small test executable, FeFET at 128KB optimized for `WriteEDP`. A good baseline to check against after a fresh Sniper run using `sniper_run_config_new/`.
- **`sniper_existing_run_1/`** — same workload and technology as above but produced using `run: existing` for both apps and tech, pulling directly from the pre-generated files in `sniper_run_config_existing/`. The model output CSV should match `sniper_new_run_1/` since it uses the same inputs.
- **`drio_new_run_1/`** — RAM run using DynamoRIO on a small test executable, FeFET at 128KB optimized for `WriteEDP`. The only RAM-schema example output, useful for comparing the cache and RAM CSV formats side by side.

## Output CSV Columns

The CSV schema differs depending on `DesignTarget`. Cache runs include hit/miss breakdowns and a cache level column that RAM runs do not have. Both schemas share a common set of workload identity, per-access tech, and workload-level model columns.

### Cache (`DesignTarget: cache`)

| Column | Source | Description |
|---|---|---|
| PatternConfig Name | apps | Pattern config JSON filename the row came from |
| Benchmark | apps | Benchmark name from profiler output |
| Profiler | apps config | Profiler used (`sniper`) |
| MemCellType | tech result | Memory cell technology (e.g. `Single-Level Cell FeFET`) |
| Cache Level | apps config | Cache hierarchy level profiled (e.g. `l2`, `l3`) |
| Design Target | system config | `cache` |
| Capacity | tech result | Array capacity including unit (e.g. `128KB`) |
| Word Width (bits) | system config | Word width used for the run |
| Optimization Target | tech result | NVSim optimization target (e.g. `WriteEDP`) |
| Total Reads | apps | Total load accesses |
| Total Writes | apps | Total store accesses |
| Total Read Latency (ms) | model | Workload-level read latency (hits + misses weighted) |
| Total Write Latency (ms) | model | Workload-level write latency |
| Total Latency (ms) | model | Sum of read and write latency |
| Total Read Energy (mJ) | model | Workload-level read energy (hits + misses weighted) |
| Total Write Energy (mJ) | model | Workload-level write energy |
| Total Energy (mJ) | model | Sum of read and write energy |
| Total Dynamic Read Power (mW) | model | Dynamic read power (read energy / execution time) |
| Total Dynamic Write Power (mW) | model | Dynamic write power (write energy / execution time) |
| Total Power (mW) | model | Leakage + dynamic read + dynamic write power |
| Read Bandwidth Usage (%) | model | Not yet implemented — placeholder |
| Write Bandwidth Usage (%) | model | Not yet implemented — placeholder |
| Cache Hit Latency (ns) | tech result | Per-access hit latency from NVSim |
| Cache Miss Latency (ns) | tech result | Per-access miss latency from NVSim |
| Cache Write Latency (ns) | tech result | Per-access write latency from NVSim |
| Cache Hit Energy (nJ) | tech result | Per-access hit dynamic energy from NVSim |
| Cache Miss Energy (nJ) | tech result | Per-access miss dynamic energy from NVSim |
| Cache Write Energy (nJ) | tech result | Per-access write dynamic energy from NVSim |
| Leakage Power (mW) | tech result | Total array leakage power from NVSim |
| Total Area (mm²) | tech result | Total array area from NVSim |
